Keycloak#

Keycloak is an open-source single sing-on (SSO) system. It is used to build authentication into information systems.

# importing some common operations
source ./keycloak_files/utils.sh

Service account#

Service account is used to configure the authentification of other services in the resources protected by Keycloak.

Check more in Using a service account sectoin of the documentation.

Users#

Keycloak handles users. The endpoint that is responsible for retrieving user information is /admin/realms/{realm}/users.

The following cell gets the id of the admin user.

access_token=$(get_admin_token | jq -r .access_token)

curl -X GET -s \
    -H "authorization: Bearer $access_token" \
    http://localhost:8080/admin/realms/master/users?username=admin | jq .[0].id

Sessions#

A session is an entity that stores user login information: the user’s identity, the time the session started and the last access time.

  • A session is created each time you request an access token. The session_state attribute of the token request contains the id of the corresponding session.

  • You can the information about sessions using following:

    • GET /admin/realms/{realm}/users/{user-id}/sessions sessions of the particular user.

    • GET /admin/realms/{realm}/clients/{client-uuid}/user-sessions sessions of the given client.

  • You can delete the given session using DELETE /admin/realms/{realm}/sessions/{sessions}.

The following cell requests an access token, which creates a new session. The next cell shows the corresponding session ID.

token_data=$(get_admin_token)
access_token=$(echo $token_data | jq -r .access_token)
session_state=$(echo $token_data | jq -r .session_state)

echo $session_state

Refresh#

There is a special mechanism to prolong the usage of the session token - refresh token. With access_token usually provided refresh_token which you can use to refresh the session.

To refresh the token use the same endpoint as to get the session but profide:

{
    "grant_type": "refresh_token",
    "client_id": {client_id},
    "refresh_token": {refresh token}
}

in the payload.

The following cell creates the session and save all the important attributes.

token_data=$(get_admin_token)
access_token=$(echo $token_data | jq -r .access_token)
session_state=$(echo $token_data | jq -r .session_state)
refresh_token=$(echo $token_data | jq -r .refresh_token)

No session#

There are plenty of cases when a session can be removed during the execution of an operation. One example is when the session is removed by the key cloak due to expiration (i.e. session expiration, not token expiration).

There are two cases:

  • The client tries to perform an operation using an invalid access token that is not valid anymore: results in a 401 response.

  • The client tries to refresh the token an invalid session: results in a 400 response.

The following cell attempts to access the KeyCloak API with an invalid token.

curl -X GET -s -w "\nStatus: %{http_code}" -H "authorization: Bearer some_access_token" \
    http://localhost:8080/admin/realms/master/users/count

{"error":"HTTP 401 Unauthorized"}
Status: 401

The following cell attempts to refresh the access token with invalid refresh token.

curl -X POST -s -w "\nStatus: %{http_code}"\
    -d "grant_type=refresh_token" \
    -d "client_id=admin-cli" \
    -d "refresh_token=$token" \
    http://localhost:8080/realms/master/protocol/openid-connect/token

{"error":"invalid_grant","error_description":"Invalid refresh token"}
Status: 400
